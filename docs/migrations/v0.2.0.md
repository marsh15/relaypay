# Migrating RelayPay v0.1.0 to v0.2.0

RelayPay v0.2.0 adds global users, organisation memberships, isolated TEST and LIVE_LIKE
environments, versioned API keys, platform administration, and immutable transactional audit
records. The migration is deterministic and does not perform network I/O.

> **Synthetic data only.** RelayPay is not a payment processor. Back up the PostgreSQL database
> before migration and never use this procedure with real financial or identity data.

## Before upgrading

1. Stop API, poller, worker, and other processes that can write RelayPay data.
2. Back up the RelayPay PostgreSQL database and record the current Alembic revision. It must be
   `0005_scenarios` for a direct v0.1.0 upgrade.
3. Retain the existing `API_KEY_PEPPER`. Changing it invalidates all migrated API keys.
4. Run the reproducible fixture proof on an empty disposable database:

   ```bash
   uv run python -m scripts.verify_m1_upgrade
   ```

## Apply and verify

```bash
uv run alembic -c migrations/relaypay/alembic.ini upgrade head
uv run alembic -c migrations/provider/alembic.ini upgrade head
uv run alembic -c migrations/receiver/alembic.ini upgrade head
uv run alembic -c migrations/relaypay/alembic.ini check
```

For each organisation, the RelayPay migration creates exactly one TEST and one LIVE_LIKE
environment. Existing financial, provider, idempotency, webhook, and scenario rows are assigned
to TEST; LIVE_LIKE starts empty. Existing users become global users with an active
`ORGANISATION_ADMIN` membership in their former organisation. Existing API keys become active
version 1 TEST keys without changing their digest, scopes, revocation state, or last-use evidence.

After upgrading, authenticate one migrated TEST key, confirm the environments endpoint returns
both environments, and run the permanent lost-response proof. New keys include an environment
marker, are displayed only once, and rotate through PENDING to ACTIVE before older active
versions are revoked.

## Operational notes

- Environment paths are organisation-scoped; cross-environment parents are rejected by composite
  PostgreSQL foreign keys and cross-boundary reads return `404`.
- Merchant references, provider stable keys, ledger account codes, refund references, and
  idempotency keys are unique within an environment rather than across both environments.
- Organisation administration requires `ORGANISATION_ADMIN`; organisation provisioning requires
  `PLATFORM_ADMIN`. Mutations use sessions, CSRF verification, and append-only audit records.
- Bootstrap the first synthetic platform administrator only through environment variables:

  ```bash
  RELAYPAY_BOOTSTRAP_ADMIN_EMAIL=platform-admin@example.test \
  RELAYPAY_BOOTSTRAP_ADMIN_PASSWORD='replace-with-a-synthetic-secret' \
  uv run python -m scripts.bootstrap_platform_admin
  ```

Downgrade is intended only for disposable local validation. Restore the pre-upgrade backup for a
production-style rollback, especially after users gain multiple organisation memberships or new
LIVE_LIKE data exists.
