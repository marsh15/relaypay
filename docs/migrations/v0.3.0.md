# Migrating RelayPay v0.2.0 to v0.3.0

RelayPay v0.3.0 adds provider statement snapshots, immutable statement imports and items,
versioned reconciliation evidence, leased run state, and append-only mismatch workflow history.
The migration is deterministic and performs no network I/O.

> **Synthetic data only.** RelayPay is not a payment processor. Back up PostgreSQL before a
> production-style exercise and never use this procedure with real payment or statement data.

## Before upgrading

1. Stop API, provider, poller, worker, and any other process that can write RelayPay data.
2. Back up the RelayPay and provider databases and record their Alembic revisions. A direct
   v0.2.0 upgrade starts at `0006_identity_environments` and `0001_provider` respectively.
3. Retain existing session, CSRF, API-key, idempotency, provider-signing, and encryption secrets.
4. Run the previous-release fixture proof in disposable schemas:

   ```bash
   uv run python -m scripts.verify_m2_upgrade
   ```

## Apply and verify

```bash
uv run alembic -c migrations/relaypay/alembic.ini upgrade head
uv run alembic -c migrations/provider/alembic.ini upgrade head
uv run alembic -c migrations/receiver/alembic.ini upgrade head
uv run alembic -c migrations/relaypay/alembic.ini check
uv run alembic -c migrations/provider/alembic.ini check
uv run alembic -c migrations/receiver/alembic.ini check
```

The RelayPay migration creates statement, run, match, mismatch, evidence-version, and workflow
history tables. The provider migration creates statement export and parsed-item tables. All new
tables start empty; v0.2 rows and immutable evidence are unchanged.

After upgrading, restart services, import one bundled synthetic provider statement into TEST,
confirm its run completes, and verify an identical source-and-byte replay returns the same import
and run. Reusing that source with different bytes must return
`STATEMENT_SOURCE_CONFLICT`. Then run the permanent lost-response proof.

## Operational notes

- Raw statement input is exact-byte evidence bounded to 1 MiB and 10,000 items.
- Imports are unique within organisation, environment, provider, and source reference. A
  transaction-scoped PostgreSQL advisory lock serializes concurrent source replays.
- Reconciliation workers reclaim expired leases with `FOR UPDATE SKIP LOCKED`. PostgreSQL is the
  only run and workflow authority; worker queues do not decide outcomes.
- `OPEN → ACKNOWLEDGED → RESOLVED` is the only mismatch transition. Evidence refresh appends a
  version. Resolution may link an existing compensation journal but creates no financial event.
- Downgrade is for disposable validation only. Restore the pre-upgrade backups for a
  production-style rollback after any v0.3 evidence or workflow rows exist.
