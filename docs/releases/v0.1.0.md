# RelayPay v0.1.0 — Proven baseline

RelayPay v0.1.0 is an evidence-first, INR-only payment-orchestration sandbox. Its primary proof
demonstrates a provider capture that commits while its response is lost: RelayPay never repeats
the mutation, recovers by signed lookup, and finalizes exactly once.

> **Synthetic data only.** This release is not a payment processor and must never receive real
> card, bank-account, identity, customer, or other regulated data. It is not deployed or hosted.

## Demonstrated guarantees

- One provider mutation for a capture even when the response is lost.
- Lookup-only recovery after a recorded send.
- Exactly one balanced, immutable capture journal and event.
- Byte-identical terminal idempotency responses for compatible capture keys.
- Signed, deduplicated webhook delivery with durable attempt evidence.
- Tenant, role, API-key scope, session, CSRF, and redaction boundaries.

## Run the demonstration

Follow the root `README.md` quickstart, sign in with a synthetic administrator, and select
**Run lost-response scenario**. The same proof is available from `make demo`. The release asset
`relaypay-v0.1.0-proof.mp4` is generated reproducibly by `scripts/video/build_release_video.sh`.

## Migration guidance

This is the first public schema baseline. Apply the RelayPay, provider, and receiver Alembic
histories to empty PostgreSQL databases, then run the synthetic seed command. Existing local
development data is disposable; reset tooling is deliberately guarded and synthetic-only.

## Known limitations

- INR-only synthetic data and deterministic mocks.
- A single merchant account per organisation in this release.
- No reconciliation, settlements, payouts, connector management, public SDK, or edge gateway.
- PostgreSQL is the financial authority; Redis/Celery only accelerates background work.
- No production SLO, PCI claim, real-money handling, deployment, or hosting.

Verification details and measured results are recorded in `docs/test-evidence.md`.
